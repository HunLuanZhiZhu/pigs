## 总览

本仓库包含 **14 个参考项目**，涵盖 5 种编程语言（另含配置/接入型仓库）：

| 语言 / 形态 | 项目数 | 项目 |
|---|---|---|
| **Rust** | 3 | claw-code, codex, grok-build |
| **TypeScript** | 6 | cline, kilocode, openclaw, opencode, pi, oh-my-openagent |
| **TypeScript + Rust** | 1 | oh-my-pi（Bun 前端 + Rust 本地核心） |
| **Python** | 2 | CoreCoder, hermes-agent |
| **Go** | 1 | deepseek-reasonix |
| **配置/接入（非完整运行时）** | 1 | fugu（Sakana 多模型编排的 Codex 接入与配置包） |

所有项目都是独立的 git 仓库（各有自己的 `.git/`），作为参考克隆到 `pigs/` 目录下，**不属于** pigs workspace 构建。grok-build 的根 `Cargo.toml` 标注为自动生成（来自 SpaceXAI monorepo 定期同步），仓库根 `SOURCE_REV` 文件记录源 monorepo 的 commit SHA。

### 共同架构主题

所有参考项目都围绕以下核心概念构建：

1. **工具调用 Agent 循环** — 每个项目都实现了某种形式的 LLM ↔ 工具 ↔ 状态机循环
2. **多供应商 LLM 抽象** — 将不同模型供应商（OpenAI、Anthropic、Google 等）抽象到统一接口后
3. **项目记忆文件** — `AGENTS.md`、`CLAUDE.md`、`REASONIX.md` 等加载到系统提示词中
4. **可扩展架构** — 插件、技能、MCP 服务器、扩展系统
5. **多前端交付** — CLI、TUI、IDE 插件、Web、桌面应用、消息平台
6. **多 Agent / 多模型编排** — fugu（协调器+模型池）、oh-my-openagent（Team Mode / 多 harness）、oh-my-pi（swarm 扩展）等把「多专家」产品化
7. **以单模型 API 交付多 Agent 系统** — fugu 将内部编排包装为 Chat Completions / Responses 单一 model 外观

### 基本信息

| 属性 | 值 |
|---|---|

| **许可证** | 见仓库 LICENSE |

### 项目说明

CoreCoder 自称是 **"coding agent 的 nanoGPT"**——一个极简的教育性 AI 编程 Agent，用纯 Python 实现。设计目标是让读者能够完整理解一个编程 Agent 的工作原理，然后 fork 它。

核心依赖：`openai`（LLM 调用）、`rich`（终端美化）、`prompt_toolkit`（交互输入）、`python-dotenv`（环境变量）。

### 架构

项目结构极简：
- `corecoder/` — 核心源代码包
- `tests/` — 测试
- `pyproject.toml` — 构建配置
- `README.md` / `README_CN.md` — 英文和中文文档

### 独特价值

- **最小可读实现**：约1k行代码实现完整的编程 Agent 循环，适合理解核心原理
- **fork 友好**：明确鼓励读者 fork 后构建自己的版本
- **对比标杆**：README 中与 Claude Code、aider、nanoGPT 做对比

### 对新 Agent 的启示

CoreCoder 提供了一个验证过的最小 Agent 循环模型。在开始 Rust 实现前，阅读其核心代码有助于理解 Agent 循环的基本结构：接收输入 → 调用 LLM → 解析工具调用 → 执行工具 → 返回结果 → 循环。

### 基本信息

|---|---|

| **构建工具** | Cargo（workspace，resolver = "2"） |
| **工作空间版本** | 0.1.3 |
| **crate 数量** | 9 个（`rust/crates/` 下） |
| **Rust LOC** | 48,599（PARITY.md 记录） |
| **测试 LOC** | 2,568 |
| **仓库地址** | github.com/ultraworkers/claw-code |
| **许可证** | MIT |
| **Lint 规则** | `unsafe_code = "forbid"` |

> ⚠️ **本项目语言为 Rust，claw-code 是最重要的参考项目之一。**

### 项目说明

claw-code 是 Claude Code（TypeScript）的 **Rust 端口**。它是一个 AI 编程 Agent CLI 工具（二进制名 `claw`/`claw.exe`），支持：
- 交互式 REPL 和一次性提示模式
- 多 LLM 供应商（Anthropic、OpenAI 兼容、xAI、Ollama）
- 完整工具系统（bash、读/写/编辑文件、grep、glob、Web 搜索/抓取、子 Agent、todo 跟踪、笔记本编辑）
- MCP 服务器生命周期管理、插件、技能、钩子、斜杠命令
- 会话恢复、成本/用量跟踪、Git 集成
- 默认模型 `claude-opus-4-7`；模型别名 `opus`/`sonnet`/`haiku`
- 默认权限 `workspace-write`

此外还有两个配套产品：
- **`claw-analog`** — 精简 Agent 循环，用于可预测的 CI/自动化流程
- **`claw-rag-service`** — 语义代码搜索服务（SQLite + OpenAI 兼容嵌入）

### Rust 工作空间架构（`rust/crates/`）

| Crate | 职责 |
|---|---|
| `rusty-claude-cli` | 主 `claw` 二进制：REPL、一次性提示、CLI 子命令、流式显示、参数解析 |
| `api` | 供应商客户端、SSE 流式、请求/响应类型、认证、请求大小预检 |
| `runtime` | `ConversationRuntime`、配置、会话、`PermissionPolicy`/`PermissionEnforcer`、MCP 客户端生命周期、系统提示词组装、bash 验证、沙箱、文件操作、任务注册表、团队 cron 注册表、MCP 工具桥接、LSP 客户端 |
| `tools` | 工具规范 + 执行：Bash、ReadFile、WriteFile、EditFile、GlobSearch、GrepSearch、WebSearch、WebFetch、Agent、TodoWrite、NotebookEdit、Skill、ToolSearch |
| `commands` | 斜杠命令注册表 + 帮助渲染 |
| `plugins` | 插件元数据、安装/启用/禁用/更新、钩子集成 |
| `telemetry` | 会话追踪、用量遥测类型 |
| `claw-analog` | 精简 Agent 循环：按模式读/搜索/写工具、TOML 配置、会话、doctor、`retrieve_context` |
| `claw-rag-service` | `ingest` + `serve` 进程；SQLite 向量；路由 `/`、`/health`、`/v1/stats`、`/v1/query` |
| `mock-anthropic-service` | 确定性 `/v1/messages` mock，用于对等测试 |
| `compat-harness` | 兼容性/对等辅助工具 |

### 设计理念（PHILOSOPHY.md）

claw-code 的 `PHILOSOPHY.md` 是其概念核心，关键要点：

1. **"停止盯着文件看"** — Rust 和 Python 代码明确被称为*副产品*。真正的研究对象是**产生它们的系统**：一个由 AI Agent 自主维护的协调循环。
2. **人类界面是 Discord** — 人类通过 Discord 给出方向，Agent 自主执行：分解任务、分配角色（架构师、执行者、审查者）、写代码、运行测试、争论失败、恢复、推送。
3. **三部分系统**：
   - **OmX**（oh-my-codex）— 工作流层：将简短指令转化为结构化执行
   - **clawhip** — 事件/通知路由器：监控 git 提交、tmux、GitHub issue/PR、Agent 生命周期
   - **OmO**（oh-my-openagent）— 多 Agent 协调：规划、交接、分歧解决、验证循环
4. **安全优先** — 相对路径、禁止 `..`、规范工作区边界、`PermissionMode`
5. **显式限制** — 读取大小、移动计数、glob/grep 上限、RAG 超时
6. **Agent 可观测性** — NDJSON 带 schema + `format_version`

### 对等状态（PARITY.md）

`PARITY.md` 是规范文件，跟踪 Rust 端口与上游 TypeScript Claude Code 的差距。9 个 lane 全部合并到 `main`：

1. Bash 验证 — `bash_validation.rs`
2. CI 修复 — `sandbox.rs` 探测 `unshare` 能力
3. 文件工具 — `file_ops.rs`（744行）：二进制检测、大小限制、工作区边界、符号链接逃逸
4. TaskRegistry — `task_registry.rs`（335行）
5. 任务接线 — `tools/lib.rs`
6. 团队+Cron — `team_cron_registry.rs`（441行）
7. MCP 生命周期 — `mcp_tool_bridge.rs`（491行）
8. LSP 客户端 — `lsp_client.rs`（461行）
9. 权限强制 — `permission_enforcer.rs`（357行）

Mock 对等测试：12个脚本场景，21个捕获的 `/v1/messages` 请求。

### 独特价值

- **与本项目语言相同** — Rust 工作空间的 crate 分层是直接的架构参考
- **安全设计范式** — 工作区边界、权限模式、显式限制，值得直接借鉴
- **模块化分层** — 9个 crate 清晰分离：api（供应商）→ runtime（状态/权限/MCP）→ tools（执行）→ cli（UX）
- **Agent 自治理念** — 虽然本项目不一定要复制 OmX/clawhip/OmO，但其 Agent 可观测性和事件优先设计值得参考

## 3. cline

### 基本信息

| 属性 | 值 |
|---|---|
| **语言** | TypeScript |
| **运行时/包管理器** | Bun 1.3.13（Node >= 22） |
| **Lint/格式化** | Biome v2.4.5 |
| **测试** | Vitest v4.0.x |
| **CLI 版本** | v3.0.39 |
| **VS Code 扩展版本** | v4.0.0 |
| **SDK 版本** | v0.0.59 |
| **仓库地址** | github.com/cline/cline |
| **许可证** | Apache 2.0 |

### 项目说明

Cline 是一个**开源 AI 编程 Agent**，同时存在于 IDE 和终端中。它是一个多面产品：同一个 Agent 引擎驱动 **CLI**、**VS Code 扩展**、**JetBrains 插件**、**Kanban 多 Agent 任务板**和 **SDK**。

核心能力：
- 跨项目读/写文件，协调多文件编辑
- 运行 bash 命令并实时监控输出
- Web 浏览（Puppeteer/Chrome）
- Plan 模式（探索+策略）或 Act 模式（执行）
- 支持所有主流 LLM 供应商（12+）
- 插件和 MCP 服务器扩展
- 多 Agent 团队协调（协调者委派给专家子 Agent）
- Cron 调度
- Slack、Telegram、Discord、Google Chat、WhatsApp、Linear 等聊天平台
- CI/CD 无头运行（JSON I/O）

### SDK 分层架构（`sdk/`）

Cline 的 SDK 采用**严格分层运行时栈**，依赖单向流动：

```
@cline/shared → @cline/llms → @cline/agents → @cline/core → Host Apps
```

| 包 | 路径 | 职责 |
|---|---|---|
| `@cline/shared` | `sdk/packages/shared/` | 低级契约、类型、schema、路径解析、钩子引擎、扩展注册表 |
| `@cline/llms` | `sdk/packages/llms/` | 供应商设置/配置、模型目录/清单、网关式供应商契约 |
| `@cline/agents` | `sdk/packages/agents/` | **无状态** Agent 循环、工具编排、运行时事件发射、钩子/扩展执行 |
| `@cline/core` | `sdk/packages/core/` | **有状态**编排：`ClineCore` 门面、会话生命周期、存储/持久化（SQLite）、配置监听、插件加载、默认工具、遥测、hub 服务器、cron/调度运行时 |

关键设计决策：
- `@cline/agents` **刻意无状态**（不关心会话/存储/宿主）
- `@cline/core` **拥有所有状态**
- `RuntimeHost` 抽象：`LocalRuntimeHost` / `HubRuntimeHost` / `RemoteRuntimeHost`

### 独特价值

- **多面架构**：同一 SDK 驱动 CLI、VS Code、JetBrains、Kanban
- **严格分层**：无状态 Agent 核心 + 有状态编排层，是值得借鉴的分层模式
- **Hub 守护进程模型**：分离的共享守护进程允许多客户端附加/分离
- **文件式自动化**：Cron 规范以 Markdown + YAML frontmatter 编写
- **多 Agent 团队**：协调者 + 专家子 Agent，状态跨会话持久化
- **上下文压缩**：会话历史保持完整保真度，压缩状态单独存储并验证

### 关键依赖

- `zod`（schema 验证）、`better-sqlite3`（会话持久化）、`simple-git`（Git 操作）、`@modelcontextprotocol/sdk`（MCP）
- `ai`（Vercel AI SDK，统一供应商接口）+ 12+ `@ai-sdk/*` 供应商包
- `@opentui/core` + `@opentui/react`（终端 UI 框架）+ `react` 19.2.4
- `commander`（CLI 参数解析）、`pino`（结构化日志）

| **许可证** | Apache 2.0 |

### 项目说明

Codex CLI 是 **OpenAI 官方的本地运行编程 Agent**。它是一个终端 AI 编程 Agent，可以：
- 本地运行交互式 Agent 循环，调用 OpenAI Responses API（和推理模型）
- 读/写文件、应用补丁、在沙箱内执行 shell 命令
- 支持多前端：TUI、`exec` 非交互模式、`app-server`（守护进程）模式、`code-mode`
- 支持多模型供应商：OpenAI/ChatGPT 登录、API 密钥、Ollama、LM Studio
- MCP 客户端和服务器
- 技能系统、记忆系统（读/写）、目标系统、可扩展插件/扩展架构

### Rust 工作空间架构（`codex-rs/`，约126个 crate）

#### 核心 Agent 逻辑
- `core/`（`codex-core`）— 中央 crate，130+源文件：`client.rs`、`codex_thread.rs`、`apply_patch.rs`、`compact.rs`（多种压缩策略）、`agents_md.rs`、`elicitation.rs`
- `protocol/` — 跨 crate 共享的协议类型
- `prompts/`（含 `templates/`）— 提示词模板
- `state/` — SQLx 迁移的持久化状态（目标、日志、记忆分别迁移）
- `thread-store/` — 会话线程存储

#### 前端
- `tui/`（`codex-tui`）— 基于 ratatui/crossterm 的终端 UI
- `cli/`（`codex-cli`）— CLI 二进制（`main.rs`、`lib.rs`，子命令：`login`、`doctor`、`mcp_cmd` 等）
- `app-server/` + `app-server-client/` + `app-server-daemon/` + `app-server-protocol/` — 程序化集成架构
- `exec/` + `exec-server/` — 非交互执行
- `mcp-server/` — 将 Codex 自身暴露为 MCP 服务器

#### 沙箱与安全（核心主题）
- `sandboxing/` — 跨平台沙箱抽象
- `linux-sandbox/` — Linux 沙箱（landlock + seccomp）
- `windows-sandbox-rs/` — Windows 沙箱
- `bwrap/` + `vendor/bubblewrap/` — Linux 容器沙箱
- `process-hardening/` — 进程加固
- `network-proxy/` — 网络代理
- `execpolicy/` — **基于 Starlark 的策略引擎**
- `shell-command/` / `shell-escalation/` — shell 命令处理和权限提升

#### 模型/供应商
- `model-provider/` + `models-manager/` — 供应商抽象
- `chatgpt/` — ChatGPT 认证/登录流程
- `ollama/` / `lmstudio/` — 本地模型集成

#### 扩展/插件/技能
- `ext/` — 扩展子 crate：`connectors`、`extension-api`、`goal`、`guardian`、`image-generation`、`mcp`、`memories`、`skills`、`web-search`
- `plugin/` / `core-plugins/` — 插件系统
- `skills/` / `core-skills/` — 技能系统
- `memories/read` + `memories/write` — 持久记忆
- `hooks/`（含 `schema/`）— 生命周期钩子

#### 工具类 crate（`utils/` 下，20+个）
`absolute-path`、`approval-presets`、`cache`、`fuzzy-match`、`home-dir`、`path-utils`、`pty`、`rustls-provider` 等——每个都是单一职责的小 crate。

### 双重构建系统

这是 codex 最架构上有趣的方面之一：

**Cargo**（`codex-rs/Cargo.toml`）：
- 主要开发构建工具
- 自定义 profile：`dev`、`dev-small`、`release`（thin LTO）、`profiling`、`ci-test`
- `cargo-shear` 元数据消除误报的未使用依赖检测

**Bazel**（`MODULE.bazel`，147个 `BUILD.bazel` 文件）：
- 用于**密封、可复现的跨平台 CI/发布构建**
- 声明依赖 `bazel_skylib`、`platforms`、**密封 LLVM 工具链**（0.7.9，带自定义补丁）
- 补丁 hermetic LLVM（3个补丁）、abseil-cpp（Windows gnullvm 线程身份）
- 从 Apple CDN 拉取 macOS SDK
- `AGENTS.md` 明确要求：修改 `Cargo.toml`/`Cargo.lock` 后必须运行 `just bazel-lock-update`

**just**（`justfile`，7.5KB）编排两者：`codex`、`exec`、`fmt`、`fix`/`clippy`、`test`、`bench`、`bazel-codex`、`bazel-lock-update`

### 关键依赖

- **异步运行时**：`tokio` 1、`tokio-stream`、`tokio-tungstenite`、`async-stream`、`futures`
- **HTTP/网络**：`reqwest` 0.12、`axum` 0.8、`tonic` 0.14（gRPC）、`rustls`
- **TUI**：`ratatui` 0.29（OpenAI fork）、`crossterm` 0.28（OpenAI fork）、`arboard`（剪贴板）
- **序列化**：`serde`、`serde_json`、`toml` 0.9、`schemars`（JSON schema 生成）、`ts-rs`（TypeScript 类型生成）
- **存储**：`sqlx` 0.9（bundled SQLite）

### Agent 运行时架构

#### 进程拓扑 — Leader/Follower IPC

运行时基于 actor 模型，每会话单线程（tokio `LocalSet` + `spawn_local`），使用 `Rc`-based `LocalRef` 模式在 `spawn_local` 任务中借用会话状态。

- 一个 **leader** 进程每台机器一份，拥有 Agent 状态并持久化到 `~/.grok/`。通过 IPC 服务器（Unix domain socket `~/.grok/leader.sock`；Windows named pipe）接受连接。
- 多个 **client**（TUI/stdio、IDE 扩展、headless CLI、websocket）通过 socket 连接，说 ACP (Agent Client Protocol) JSON-RPC。leader 命名空间化 request ID 并跟踪会话所有权。
- `connect_or_spawn` 要么附加到已存在的 leader，要么启动一个。定期 `run_auto_update_checker` 可以 flush 会话 actor 并重启 leader 以应用更新后的二进制。
- 三个入口：`run_stdio_agent`、`run_headless`、`run_leader`。

#### SessionActor — 核心状态机

每个会话运行在自己的线程上，拥有：
- mpsc 命令通道（`SessionCommand`：Prompt、Initialize、SetSessionModel、RebuildAgentForDefinition、SessionMode、后台任务控制等）
- 事件通道 + chat-state 事件通道 + completion 通道
- `Agent`（通过 `AgentBuilder` 构建一次）、`ToolBridge`、`ChatStateHandle`、`HunkTrackerActor`、MCP 状态、memory 状态、hooks registry、permission handle、observability bridge

主循环 `run_session` 是 biased `tokio::select!`：
- idle memory flush / dream-check 定时器
- model-switch watch channel
- chat-state 事件（会话重置、image-budget）
- 会话事件（通知、replay flush）
- **completion 通道** — 回合完成时调用 `handle_completion` → `handle_turn_end` → 排空 interjections/notifications → `maybe_start_running_task` 执行下一条排队 prompt
- **command 通道** — 分发 `SessionCommand`。通道关闭时触发 `SessionEnd`/`Stop` hooks，运行 memory `on_session_end`，持久化后台任务，退出。

#### 单个 Agent 回合

`process_conversation_turn` 中的 `loop { }`：
1. emit `LoopStarted`，排空 interjections，flush skill reminders，注入 monitor 事件，首轮 memory reminder，MCP reminder。
2. 启用则预执行 two-pass compaction；超过阈值则自动压缩。
3. 解析有效工具（`Agent::tool_definitions()` 基础 specs + fork 覆写；JSON schema 请求注入合成 `StructuredOutput` 工具）。
4. `chat_state_handle.build_request(...)` 构建请求；tag `x-grok-session-id`、`x-grok-turn-idx`、`x-grok-agent-id`、`x-grok-deployment-id`。
5. `run_turn_via_sampler(request)` 流式响应，带 auth-retry backoff、`CompactAndResubmit`、`RefreshAuthAndResubmit`。
6. 记录 assistant items；推送 tool 结果到 chat state。
7. 无工具调用 → **TodoGate**（推动模型推进 pending todos，bounded `max_fires_per_prompt`）→ 排空 interjections → finalize → break。
8. 有工具调用 → `execute_tool_calls` → 推送结果 → 继续循环。

### 工具系统

#### 统一 Tool trait

```rust
pub trait Tool: Send + Sync {
    type Args: Deserialize<'de> + JsonSchema + Send + 'static;
    type Output: Serialize + ToolOutput + Send + 'static;
    fn id(&self) -> ToolId;
    fn description(&self, _ctx: &ListToolsContext) -> ToolDescription;
    fn capabilities(&self) -> ToolCapabilities { default }
    fn should_list(&self, _ctx: &ListToolsContext) -> bool { true }
    fn execute(&self, ctx, args) -> impl Future<Output = ToolStream<Self::Output>> + Send;
    fn run(&self, ctx, args) -> impl Future<Output = Result<Self::Output, ToolError>> { NotImplemented }
}
```

`ToolStream<T>` 是 pinned stream，不变量：多个 `Progress` item 以唯一一个 `Terminal` 结束。`ToolDyn` 是类型擦除形式。工具覆写 `run`（阻塞）或 `execute`（流式），运行时始终调用 `execute`。

#### Registry & Bridge

- `ToolRegistryBuilder` → `FinalizedToolset`：进程级 `register_tool_pack(fn(&mut ToolRegistryBuilder))` 允许外部工具包在启动时注入。
- `ToolBridge` 包装 `FinalizedToolset` + 独立存储的 `Terminal`（避免取消时死锁）。
- `ToolConfig`：id、params、name_override、behavior_version、kind。
- `Resources`/`SharedResources`：类型化 map（`Terminal`、`Cwd`、`ToolCallId`、`SessionFolder`、`SessionEnv`、`NotificationHandle`、`TruncationCfg`、`TemplateRenderer`、`Params`）。

#### 执行路径

- `execute_tool_calls`：接收 `Vec<ToolCallResponse>`，per-call 调用 `prepare_tool_call`（返回 `PreparedToolCall` 或拒绝：`PermissionReject`/`Cancelled`/`FollowupMessage`/`HookDenied`）。批准的调用计算 per-file write locks（同文件编辑在 `tokio::Mutex` 下按模型输出顺序串行；读操作并发）。
- `dispatch_tool`：调用 `WorkspaceOps::call_tool`。Agent 会话始终使用本地 workspace ops（进程内 toolset）。
- `call_with_auth_retry`：401 时触发 `AuthManager::try_recover_unauthorized`（通过 `OnceCell` 去重并发 401）并重试一次。

内置工具实现位于 `implementations/grok_build/`，包括 bash、grep、read_file、search_replace/hashline、list_dir、todo、task/subagent、web_search/fetch、image_gen、video_gen、monitor、scheduler、ask_user_question、kill_task、lsp、enter/exit_plan_mode、update_goal、deploy_app。

### TUI 架构

基于 **ratatui** + **crossterm** + **syntect** + **ansi-to-tui** + **textwrap** + vendored Mermaid stack。

pager 是 leader 的 *client* — 通过 `connect_or_spawn` 连接并通过 leader socket 交换 ACP JSON-RPC（或单进程模式下走 stdio）。

`app/mod.rs` 文档化架构：
- **actions** — `Action`、`Effect`、`TaskResult` 枚举
- **agent / agent_view** — `AgentSession`/`AgentId`/`TurnState` 业务类型 + `AgentView` per-agent view-model
- **dispatch** — `Action → 状态变更 + Vec<Effect>`（同步、可测试）
- **effects** — `Effect → async task spawning`
- **event_loop** — biased `tokio::select!`
- Off-thread workers：`edit_highlight_worker`（syntect 编辑 diff）、`mermaid_worker`（进程外 Mermaid 渲染 + 缓存）、`foreign_sessions`（扫描外部 Claude/Codex 会话）、`roster`、`status_blocks`、`turn_completion`、`subagent`、`subscription`、`leader_cluster`

终端管理使用 crossterm（`EnterAlternateScreen`、`SetTitle`、`SetCursorStyle`、`EnableMouseCapture`、bracketed paste、Kitty keyboard enhancement）。

### 子智能体 / 多 Agent 支持

一等公民，非常完善（`agent/subagent/` + `mvp_agent/subagent_coordinator.rs`）。模型通过 **task 工具**生成子会话。

- `SubagentCoordinator`（存储在 `MvpAgent` 上）管理三个 map：`pending`（初始化中）、`active`（运行中 `SubagentTracker`）、`completed`（保留供 `TaskOutputTool` 轮询和 `resume_from`）。`completion_notify: Arc<Notify>` 支持多等待；`running_gauge` 馈入 `AgentActivity::is_busy`。
- `SubagentSpawnContext`：约 370 字段的参数包，携带子会话继承的一切：父的 LSP、fs、terminal runner、notification handle、scheduler handle、session env、hunk tracker、memory config、web search/fetch/image/video/deploy configs、sampling config、auth、MCP configs + managed-MCP state + MCP pool snapshot、skills、personas、roles、permission handle、workspace ops、hook registry。子会话共享后端，因此 edits/bash/reads 走同一终端。
- **模型解析**：runtime override > `config.toml [subagents.models]` > `AgentDefinition.model = Override(id)` > 继承父的 live sampling config。
- **初始上下文**：resume-from-peer（fail-closed，复制源 transcript + tool state，80% 子上下文窗口门控）> live fork（字节级镜像，保留缓存）> New。
- `run_in_background` 子智能体异步完成；阻塞 spawn 持有 `BlockingWaitGuard`。自动唤醒将完成作为合成 prompt 注入。
- 子智能体可使用 git worktree 隔离（`isolation=worktree`），退出时可选 durable ref 快照。

还有 `/goal` 自主编排层（`goal_orchestrator.rs`、`goal_planner.rs`、`goal_strategist.rs`、`goal_classifier/`、`goal_stop_detector.rs`、`goal_summarizer.rs`）和 `laziness`/`laziness_classifier` 子系统推动停滞循环。

### MCP 支持

`xai-grok-mcp` 刻意**隔离** `rmcp` 2.1 + `reqwest` 0.13（workspace 其余在 reqwest 0.12），避免版本升级级联。

- 传输：rmcp `StreamableHttpClientTransport` + `TokioChildProcess`
- 凭据：磁盘 `$GROK_HOME/mcp_credentials.json` + rmcp `CredentialStore`
- OAuth：浏览器 OAuth with cross/in-process dedup
- managed-MCP 刷新、liveness watcher、auto-restart
- 工具命名空间：`server__tool`（`MCP_TOOL_NAME_DELIMITER`）

支持 managed MCPs、plugin-provided MCPs、client-registered MCPs（via ACP `McpServer`）。子智能体继承父的 MCP pool snapshot。

### Hooks / Plugins 系统

**Hooks**（`xai-grok-hooks`）：文件式、命令执行、fail-open 默认。从 `~/.grok/hooks/` 和 `<git-worktree-root>/.grok/hooks/` 发现 JSON 文件。

事件类型：`SessionStart`、`PreToolUse`（阻塞——可 deny/allow）、`PostToolUse`、`PostToolUseFailure`、`SessionEnd`、`Stop`、`SubagentStart`、`SubagentStop`/`SubagentEnd`。

**Plugins**（`xai-grok-plugin-marketplace`）：marketplace 浏览/扫描/安装系统。git-install 和 local-path 插件，`PluginScope`。官方源 `https://github.com/xai-org/plugin-marketplace.git`。插件可贡献 agents、MCPs、tools、skills、rules。

### 沙箱 / 权限

**沙箱**（`xai-grok-sandbox`）：OS 级沙箱 via `nono`（Linux Landlock, macOS Seatbelt）。进程启动时应用一次；覆盖进程内 `tokio::fs` 和子进程。网络在进程级开放（Agent 需要 LLM API），但**子进程网络通过 seccomp per-subprocess 阻断**。

**权限**（`xai-grok-workspace/src/permission/`）：`PermissionManager` actor。决策流：`CompiledPolicy` → `resolution` → `rules` → `auto_mode`（LLM/启发式分类器，支持 YOLO/auto-approve）。`AcpPrompter` 渲染 ACP 权限提示。支持远程/HITL 权限 via hub。`PermissionMode` on `AgentDefinition` 选择 plan/auto 等。

### 会话管理

- `xai-chat-state` 拥有规范对话（`ChatStateHandle`、`ChatStateEvent`、采样配置、压缩 transcript、image budget、prompt index、model metadata）
- 每会话目录 `~/.grok/sessions/<encoded-cwd>/<uuid>/`：`summary.json`、`updates.jsonl`（持久化 ACP notification 流）、chat history JSONL
- 支持 session fork/merge（子智能体和 rewind）、rewind、replay events（buffered notification replay with flush timer）
- auto-compaction with configurable threshold percent、two-pass prefire
- `session/memory/`：跨会话记忆存储 with idle flush + "dream" consolidation
- `prompt_queue.rs`：prompt 排队
- `interjection.rs`：回合中注入用户消息

### LLM API 协议

多后端。`xai-grok-sampler` 的 HTTP 客户端支持三种后端形态：
- **Chat Completions**（`/chat/completions`）
- **OpenAI Responses API**（`/responses`）— via `async_openai::types::responses`
- **Anthropic Messages API**（`/messages`）

`ApiBackend` 选择使用哪个。`SamplerConfig` 携带 `base_url`、`api_key`、`model`、`max_completion_tokens`、`temperature`/`top_p`、`auth_scheme`、`extra_headers`、`context_window`、`reasoning_effort`、`api_backend` 等。

三层采样 API：
1. **Layer 1**：`SamplingClient` 返回原始 chunk 流（SSE via `eventsource_stream`）
2. **Layer 2**：`stream` 转换为 `SamplingEvent`s — `stream_chat_completions`、`stream_responses`、`stream_messages`
3. **Layer 3**：`SamplerHandle`/`SamplerActor` 管理并发请求 with retry、cancellation、`DoomLoopSignalCollector`（检测卡住循环）、per-request `x-grok-*` headers

**托管/后端执行工具**：`Agent` 携带 `hosted_tools`（如 `WebSearch`、`x_search`），作为原生 Responses API 类型发送并由服务端 agentic sampler 执行，gated by `backend_search_enabled` AND per-model `supports_backend_search`。支持 structured output（原生 `request.json_schema` + 合成 `StructuredOutput` 工具两条路径）。

### 对新 Rust Agent 的设计启示

基于本项目语言为 Rust 的决定，以下是最值得借鉴的设计模式：

#### 1. 工作空间分层（参考 claw-code + codex）

推荐的 crate 分层方向：
```
api/          → LLM 供应商客户端、流式、认证
runtime/      → Agent 循环、状态、权限、会话
tools/        → 工具规范 + 执行（bash、文件操作、搜索等）
cli/          → CLI 二进制、TUI、参数解析
plugins/      → 插件系统、MCP 桥接
telemetry/    → 追踪、用量遥测
```

grok-build 的分层进一步验证了这一方向，且展示了更细的拆分：`shell`（运行时）、`agent`（Agent 定义+构建器）、`tools`（工具实现）、`tools-api`（protobuf 线类型）、`tool-runtime`（统一 trait）、`tool-protocol`（gRPC 协议）、`workspace`（文件系统+VCS+权限）、`sampler`（LLM 客户端）、`mcp`、`hooks`、`sandbox`、`config`、`memory` 等各自独立。约 84 个 crate 的拆分粒度介于 claw-code（9 个）和 codex（~126 个）之间。

#### 2. 沙箱与安全（参考 codex + claw-code + grok-build）

- 工作区边界强制执行（claw-code 的相对路径、禁止 `..`）
- 类型化权限模式（`PermissionMode`）
- 跨平台沙箱抽象（codex 的 `sandboxing/` crate）
- 显式限制（读取大小、glob/grep 上限、超时）
- OS 级沙箱 + per-subprocess seccomp 网络阻断（grok-build 的 `nono` + 子进程网络阻断）——pigs 可在 bash 工具中参考此模式限制子进程网络

#### 3. 多供应商 LLM 抽象（参考 codex + cline + pi + oh-my-pi + grok-build）

- 统一接口抽象不同供应商（OpenAI、Anthropic、Google、本地模型）
- 流式响应处理（SSE）；OpenAI 侧区分 **Responses** 与 **Chat Completions**
- 模型目录/清单管理（context window、能力标签）
- 供应商特定认证流程
- **grok-build 的三层采样 API**（Layer 1 原始流 → Layer 2 事件转换 → Layer 3 actor 管理并发+重试+取消）是比 pigs 当前单层客户端更成熟的架构；**doom-loop 检测**也值得借鉴

#### 3.5 多 Agent / 单模型 API（参考 fugu + oh-my-openagent + grok-build）

- **Fugu 形态**：内部协调器 + 模型池，外部一个 model id + OpenAI 兼容 API
- **OmO Team Mode**：lead + 并行成员 + 任务工具；工作流入口（如 ultrawork）
- **grok-build 子智能体**：`SubagentCoordinator` 管理 pending/active/completed 三态；`SubagentSpawnContext` 继承 370+ 字段；上下文 fork 策略（live fork / resume-from-peer / new）；模型解析链（runtime override > config > AgentDefinition > 继承父）；worktree 隔离
- pigs 可渐进：experts 配置 → Trinity/Council 编排 → `pigs-server` 暴露为 `model=pigs-fugu`

#### 4. 项目记忆文件（参考所有项目）

- 实现 `AGENTS.md` 加载到系统提示词
- 支持项目级和用户级的记忆文件层级
- **grok-build 的 `xai-grok-memory` crate**：跨会话记忆存储 + idle flush + "dream" 整合（idle 时自动整合记忆），是比 pigs 当前 `memory.md` 更主动的记忆系统

#### 5. 工具系统（参考 codex + claw-code + kilocode + grok-build）

- 每个工具定义为规范（schema）+ 执行（implementation）对
- 工具注册表模式
- 核心工具集：bash、文件读/写/编辑、grep、glob、Web 搜索/抓取
- **grok-build 的 `Tool` trait + `ToolStream<T>`**：工具返回流式输出（多个 Progress → 一个 Terminal），比 pigs 的 `ToolResult` 更适合长运行工具（如 bash 实时输出、web 搜索进度）
- **grok-build 的 `SharedResources` 类型化 map**：用类型化 map 传递 `Terminal`/`Cwd`/`SessionFolder` 等资源，比 pigs 逐参数传递更可扩展
- **grok-build 的进程级 `register_tool_pack`**：允许外部注册工具包，比 pigs 编译时注册更灵活

#### 6. 可观测性（参考 codex + claw-code + grok-build）

- NDJSON 机器可读输出（带 schema 版本）
- OpenTelemetry 集成
- 会话追踪和用量遥测
- `doctor` 健康检查命令
- **grok-build 的 per-request `x-grok-*` headers**（conv-id、req-id、session-id、turn-idx、agent-id、deployment-id）是请求追踪的好实践

#### 7. 模块化原则（参考 codex + grok-build）

- "优先添加新模块而非增长现有模块"（codex AGENTS.md）
- 单一职责的小 crate
- 严格的 Clippy lint（禁止 `unwrap_used`、`expect_used`）
- **grok-build 的依赖隔离**：MCP crate 隔离 `rmcp` 2.1 + `reqwest` 0.13，避免版本升级级联影响整个 workspace——pigs 可参考此策略隔离有版本冲突风险的依赖

#### 8. 进程拓扑与多前端（参考 grok-build + codex）

- **grok-build Leader/Follower IPC**：单进程 leader 拥有状态并持久化，多 client（TUI/IDE/headless）通过 ACP JSON-RPC 附加/分离。pigs 当前的 proxy+CLI 模式可以演进为此架构：API 服务器作为 leader，外部 CLI/IDE 通过标准协议连接
- **ACP (Agent Client Protocol)**：grok-build 通过 ACP 协议支持 IDE 嵌入。pigs 评估是否实现 ACP 以被外部 IDE/工具嵌入
- **codex 多前端**：TUI + exec + app-server + MCP-server，一个核心多个前端

### 不建议直接复制的方面

- codex 的 Bazel 构建系统（过度工程，除非有类似 CI 规模需求）
- claw-code 的 OmX/clawhip/OmO 自治协调系统（理念有趣但超出初始范围）
- openclaw 的 150+ 扩展（初始阶段不需要如此大的扩展面）
- hermes 的六种终端后端（初始阶段 local 执行足够）
- **Fugu 的闭源协调器训练**（学形态，不抄权重）
- **OmO 的全量 hooks/agents 矩阵**（先子集：team/council + 关键 hooks）
- **grok-build 的 leader/follower IPC**（初期 proxy+CLI 足够，但留好演进路径）
- **grok-build 的 `SubagentSpawnContext` 370+ 字段**（学三态管理和上下文 fork 策略，不抄参数爆炸）
- **grok-build 的 vendored Mermaid stack**（TUI 内渲染图是锦上添花，初期不需要）
