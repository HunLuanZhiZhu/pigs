# 术语规定
1. pigs-proxy -> 路由层， 主要负责对agent的api请求进行重试， 在本项目中还负责将虚拟的 `X-pig` 模型交给相位层进行处理
2. pigs-api -> 相位层，是本项目的核心部分；目的是为了在API层注入 1.规划；2.目标；3.失败重试 的能力

# 当前问题
## 1. 丢失信息
pigs-proxy 发给 pigs-api 的东西是自定义的，不是原始的HTTP请求，
## 2. 接口单一
当前只有 openai-chat 协议的支持；需要支持完整版的pigs-proxy的三种协议；
## 3. 流程不清
当前流程和本设计文档不一致


# 设想流程
## 总流程
上游agent的api请求 -> 注：包含历次交互和其他的
路由层 -> 注：需要判断是否是 -pig 的虚拟模型，如果不是，路由层内部处理；如果是，应该将HTTP请求完整的信息交给相位层进行处理
相位层 -> 注：并且要支持将不同相位的输出进行协调，包装为连续的输出，并且需要支持流式输出
路由层 -> 注：相位层改造完HTTP请求后，通过HTTP协议重新发给路由层（此时的模型名已经去掉 -pig） 注：本项想走内部接口的，但是觉得这样逻辑更清晰
下游模型提供商的api响应
## 相位层内部流程
执行前 -> 判断为简单任务直接结束/非简单任务执行时
执行时 -> 执行任务
执行后 -> 对任务的执行情况进行评判
分为3种情况：
1. 判定目标是否完全完成；仅当任务完成时结束（输出标记）。
2. 判定任务的执行是否是错误的，当任务执行是错误的，重新回到执行前（输出标记）；
3. 不再以上两条，也就是模型进行了推进，但不完全完成任务，应该继续执行（不用转给执行时）。
结束后，发现没有输出标记，自动重进执行后相位（保留执行时和上N次执行后的执行轨迹）

# 提示词
三个阶段都只修改**用户本次提问**的提示词，平且拼接在用户的原始提问后面。不修改其他任何提示词，也不修改请求的其他字段
约定称呼
1. 用户提问前为历史消息，也就是说在三个阶段中历史消息指的范围不变。
2. 每个相位中，不断调用tool直到结束的过程称呼为本相位执行轨迹。

## 执行前
总输入：历史消息+拼接后的提示词

以下是拼接后的提示词
```
{用户的提示词，完全保持原样，目的是：1. 保持用户的原始需求，2. 对模型的缓存命中更友好}

---

以上是本任务要求
{（如果已经有失败）
这个任务执行中曾失败过
第1次失败：
{失败时的，执行后完整输出}
...
第{n}次：
{失败时的，执行后完整输出}
...}
本次回答不需要具体执行任务，而是对任务进行执行前分析，严格按要求输出这5个问题的回答
1）本任务需要哪些项目内部信息？（指的是项目文件信息，，需要的文件路径或者关键词或者其他查找方式）
2）本任务需要哪些项目外部信息？（如 1.本机信息 2.网络信息 3.用户补充 4.模型知识）
3）为本任务的执行进行制定执行计划
4）总结本任务的目标（goal），用于判断任务是否完成
5）判断本任务是否为简单问题。当且仅当1）-4）均不需要或极其简单时，才判定为简单问题，并必须给出理由，然后在回答最后单独一行输出 PIGEND。若不是简单问题，不要输出 PIGEND
```
## 执行时
总输入：历史消息+拼接后的提示词
（也就是说不包括执行前执行轨迹，执行前输出的结果通过拼接的方式进入提问，执行时看不到执行前的提示词）

以下是拼接后的提示词
```
{用户的提示词，完全保持原样}

---

以下是对本次任务在 1.内部信息的需求 2.外部信息的需求 3.执行计划 4.任务目标 5.任务难度方便的分析：
{执行前的完整输出}
完成对内部信息和外部信息的获取后，按照执行计划全力完成任务目标
```

## 执行后
总输入：历史消息+执行时的轨迹（如果是多次进入执行后也保留每次的）+无拼接的提示词

以下是提示词：
```
根据设定的目标，独立验收结果，对任务的执行情况进行全面审查。
先输出评审报告，进行如下判断：
1. 判定目标是否完全完成。仅当任务完成时，在回答最后输出 `PIGEND`
2. 判定任务是否是执行错误的，也就是当前离目标的距离比执行前更远，在回答最后输出 `PIGFAIL`
3. 如果是推进了任务，但没有完全完成任务；在输出评审报告后，继续执行任务，最后不输出标记
```


## 提示词解释
1. 之所以不出现 “我” “用户” 这样第一或第三人称，是因为，采用中性的 “任务”
2. PIGEND 必须给出理由，不应该轻易输出
3. 不出现我们PIGS本身的信息，例如“你是PIGS相位层执行阶段”，会导致模型的身份混乱加重模型的负担
4. 提示词设计的目的是最大程度降低对模型输出格式化的依赖形成较强的鲁棒性

# 已确认实施决策（2026-07-14）

以下内容为本次实施时确认的产品语义，追加记录，不改写前述原始需求：

1. **语义完整而非字节相等**：`-pig` 请求以方法、path/query、raw header value 和完整 JSON body 进入相位层；允许 JSON 重新序列化，不要求空白、字段顺序或字节完全一致。
2. **定点修改**：每个相位 clone 原 body，只修改 model、当前用户输入和相位轨迹。原 system/instructions、历史、tools/tool_choice、推理与缓存字段、metadata、媒体块及未知扩展字段保持原生结构。
3. **三协议原生**：OpenAI Chat Completions、Anthropic Messages、OpenAI Responses 输入什么协议就输出什么协议，非流式与 SSE 均遵循入口协议。
4. **HTTP loopback**：相位 subrequest 去掉一层 `-pig` 后，通过带随机内部令牌的本机 HTTP 重新进入 pigs-proxy，继续执行渠道选择、模型映射、body 清洗、思考强度注入与同渠道重试。内部 header 在发往供应商前移除。
5. **严格状态机**：`Pre -> Executor -> Post`；Post `PIGEND` 成功完成，`PIGFAIL` 回 Pre，无标记继续 Post。只有完成标记产生成功终态，预算耗尽返回错误。
6. **标记规则**：只接受最后一个有效非空行的 `PIGEND/PIGFAIL`，且标记前必须有理由或正文。所有控制标记行对外隐藏，普通句子中出现同名词不删除。
7. **提示词**：相位指令只进入 user payload，不注入 Pigs 身份 system prompt。Pre 保留编号失败记录；Executor 获得完整 Pre 输出；Post 使用独立验收提示，并保留 Executor 与历次 Post 的协议原生轨迹。
8. **上游工具执行**：原请求 tools 不在服务端执行。模型返回工具调用时，运行时暂停并按入口协议原样返回；下一次带原生 tool result 的 `-pig` 请求按 tool-call ID 恢复同一相位。
9. **有界 continuation**：continuation 只存在进程内存，具有 TTL、容量上限和有界失效墓碑；重启、过期、淘汰、重复消费、未知 ID、错协议或错模型均返回明确错误。认证 headers 不写入 continuation，恢复时使用当前请求 headers。
10. **连续全部输出**：Pre、Executor、每次 Post 的可见文本按执行顺序连续输出并聚合 usage。内部 SSE 使用末行缓冲，已确定安全的文本立即转发；错误流只发送协议原生 error/failed，不发送成功终止帧。
11. **CLI 暂不迁移**：HTTP 路径使用新的 `HttpPhasedRuntime + LoopbackPhaseTransport`；CLI 继续保留 `ProxyApiClient / dispatch_in_process / PhasedRuntime`，本次不调整 CLI 展示、会话与本地工具循环。
