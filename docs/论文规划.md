# pigs 论文规划

## 论文1：系统设计论文

### 定位
系统设计论文，介绍 pigs 的相位编排架构。投稿目标：系统类会议或 workshop（如 EuroSys/SOSP workshop、ACM SIGOPS、或 arXiv 预印本）。

### 核心贡献
1. **相位编排状态机**：Pre→Executor→Post 三阶段编排，通过文本控制标记（PIGEND/PIGFAIL）实现路由，不依赖工具调用做 handoff
2. **协议原生 HTTP 请求保真**：完整保留客户端的 HTTP 方法、路径、headers 和 JSON body，只定点修改 model、当前 user 输入和相位轨迹
3. **HTTP loopback transport**：相位子请求通过本机 HTTP 重入代理，复用渠道选择、模型映射、body 清洗和重试逻辑
4. **有界内存 continuation**：工具调用暂停/恢复机制，受 TTL 和容量限制，不写磁盘
5. **双 transport 架构**：HTTP 路径走 loopback，CLI 路径走 HTTP 端口，共享相位提示词和标记规则

### 论文结构
1. **Introduction**：LLM Agent 编排的挑战——单次调用缺乏规划/验收能力，多 Agent 系统过重
2. **Background & Motivation**：ReAct/Reflexion/Self-Refine 的局限性；LLM middleware 的学术空白
3. **System Design**：
   - 3.1 架构概览（pigs-proxy + pigs-api + pigs-cli 三层）
   - 3.2 相位状态机（Pre→Executor→Post + PIGEND/PIGFAIL/无标记）
   - 3.3 协议原生请求信封（HttpRequestEnvelope）
   - 3.4 HTTP loopback transport
   - 3.5 有界内存 continuation
   - 3.6 连续 SSE 输出与标记隐藏
4. **Implementation**：Rust workspace 13 crate 分层；三协议 codec；标记检测算法
5. **Related Work**：与 ReAct/Reflexion/Self-Refine 对比；与 XFlow/LLM-as-Code/Compile-Then-Page 对比；与 LLM middleware (TokenMizer/STREAM) 对比
6. **Discussion**：控制标记的鲁棒性；双重编排问题；CLI 与 HTTP 路径统一
7. **Conclusion**

### 关键设计决策记录（论文中需阐述）
- 为什么用文本标记而不是 tool/MCP handoff（鲁棒性、不注入编排用 tool）
- 为什么 Post 无标记继续 Post 而不是回 Executor（避免无限回环）
- 为什么相位指令只进入 user payload（保持 system 原样、缓存友好）
- 为什么 HTTP loopback 而不是进程内调用（逻辑清晰、复用代理逻辑）

### 相关工作对比表
| 系统 | 编排方式 | 控制信号 | 协议支持 | 工具执行 |
|---|---|---|---|---|
| ReAct | 交错推理+行动 | 无（靠 LLM 自行决定） | 单协议 | 内部 |
| Reflexion | 反思+重试 | 语言反馈 | 单协议 | 内部 |
| Self-Refine | 生成+批评+改进 | 语言反馈 | 单协议 | 无 |
| pigs | Pre→Executor→Post | PIGEND/PIGFAIL 标记 | 三协议原生 | 外部 Agent |

## 论文2：实验研究论文

### 定位
基于论文1的系统做 benchmark 实验，验证相位编排对任务完成质量的影响。不需要 GPU，只需要 API 调用。

### 可用公开 benchmark 平台

| Benchmark | 任务类型 | 公开可用 | 适合验证什么 |
|---|---|---|---|
| **SWE-bench** | 软件工程（修 GitHub issue） | ✅ github.com/princeton-nlp/SWE-bench | 相位编排对代码任务的效果 |
| **AgentBench** | 8 种环境（OS/DB/Web 等） | ✅ github.com/THUDM/AgentBench | 通用 Agent 能力 |
| **GAIA** | 466 个真实世界问题 | ✅ 300 题公开 | 工具使用+推理 |
| **ToolBench** | 16464 个 REST API | ✅ github.com/OpenBMB/ToolBench | 工具调用准确性 |
| **API-Bank** | 73 个 API + 314 对话 | ✅ | API 调用+规划 |
| **BFCL** | 函数调用准确性 | ✅ gorilla.cs.berkeley.edu | 工具调用精度 |
| **AgentBoard** | 9 种多轮任务 | ✅ | 进度率指标 |
| **tau-bench** | 航空/零售客服 | ✅ | 多轮工具交互 |
| **MCP-AgentBench** | 33 个 MCP 服务器 | ✅ | MCP 工具评估 |

### 推荐实验设计

**实验1：相位编排 vs 无编排（消融）**
- Benchmark: SWE-bench Lite (300题) 或 GAIA (300题)
- 对比条件：
  - A: 直接调用（无相位编排，passthrough）
  - B: Pre→Executor→Post（完整相位编排）
  - C: 仅 Pre（规划但不验收）
  - D: 仅 Executor→Post（执行+验收但不规划）
- 指标：任务完成率、平均轮次、Token 消耗、延迟

**实验2：控制标记的有效性**
- Benchmark: GAIA 或 AgentBoard
- 对比条件：
  - A: 有 PIGFAIL（失败重规划）
  - B: 无 PIGFAIL（失败不重规划，直接结束）
- 指标：错误恢复率、重规划后成功率

**实验3：三协议一致性**
- Benchmark: BFCL 或 ToolBench
- 对比条件：
  - A: OpenAI Chat 协议
  - B: Anthropic Messages 协议
  - C: OpenAI Responses 协议
- 指标：三协议输出一致性、工具调用准确性差异

**实验4：成本-效益分析**
- Benchmark: SWE-bench Lite
- 指标：每任务 Token 消耗、API 调用次数、延迟、成功率
- 对比：相位编排带来的额外成本 vs 质量提升

### 实验执行方式
- 所有实验通过 pigs 的 HTTP API 执行（`-pig` 模型走相位编排，非 `-pig` 走透传）
- 不需要 GPU——只需要 LLM API 配额
- 可用 zcode/其他 Agent 作为外部工具执行器
- 结果自动收集到 JSONL，后续分析
