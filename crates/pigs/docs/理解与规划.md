# Pigs：理解与规划

> 本文记录对 **pigs**（相位化 Agent 运行时 / API 向）的当前理解与实现规划。  
> 与本地 REPL 宿主 **pigs-cli** 区分：二者共享 workspace 内多个库 crate，职责不同。  
> 日期：2026-07-12（随讨论迭代）

---

## 1. 我们在答什么题

| 参照 | 它们答的题 | 我们不照抄的原因 |
|---|---|---|
| **河豚（Fugu）** | 多 frontier **模型池** + 编排，对外一个 model API | 我们没有自有模型池与编排训练；不应假装小河豚 |
| **专家团（Council / Team）** | 多角色 / 多 agent 协作流程 | 可作后期可选模式，不是第一天内核 |
| **本机 coding agent（Codex 等）** | 真 shell + 真工具链 + 常配 OS 沙箱 | 初期目标是 **信息搜集向 API**，不是全能本机 IDE agent |

**我们的命题（当前共识）：**

> 在「拼消息 → 调模型」这一层做手脚，做成 **同模型、可流式** 的相位机：  
> **执行前（Plan）→ 执行者（信息向执行）→ 执行后（Review + Goal）**；  
> 把 **Plan / 信息搜集 / Goal 检** 做成 API 内置能力；  
> 编排控制面 **不靠 tool/MCP handoff**，只靠 **两标记 + 默认边**；  
> 阶段结束靠 **tool loop idle（不再产出 tool）**。

对外可以包装成 OpenAI 兼容的「一个 model / 一次流」；对内是明确的状态机。

---

## 2. 产品入口：唯一二进制 `pigs`

| 模式 | 命令 | 作用 |
|---|---|---|
| **默认 API（对外）** | `pigs` / `pigs --port 3927` | 绑定本机端口；HTTP 只是传输层 |
| **CLI（对内）** | `pigs --cli` | **不绑端口**；终端交互 |
| **单次回合** | `pigs --once "…"` | 不启 HTTP，直接转化一轮 |
| **元信息** | `pigs --describe` | 打印架构说明 |

### 2.1 共享核心（必须复用）

```text
OpenAI-shaped messages / 用户输入
        │
        ▼
   api_convert   ← 唯一「API 转化」逻辑（解析 messages → history+question）
        │
        ▼
   PhasedRuntime ← Pre → Executor → Post
        │
   ┌────┴────┐
   ▼         ▼
 HTTP/SSE   CLI/once   （仅 transport 不同）
```

`pigs-cli` **不是**产品 bin，只保留为 legacy **library**（旧 agent 模块），用户不可见。

### 2.2 本地 API 路由

- `GET /health`
- `GET /v1/models`
- `POST /v1/chat/completions`（`stream=true|false`）

默认只绑 loopback。

## 3. 核心机制理解

### 3.1 两层正交概念

| 问题 | 判据 |
|---|---|
| **本阶段是否结束？** | 模型本拍 **不再输出 tool_call**（相位内 tool loop idle） |
| **下一阶段去哪？** | 本阶段 **最终文本** 是否匹配到合法标记 |

- 标记 **不负责** 宣布阶段结束。  
- 无 tool = 收工；标记 = **路由**。

### 3.2 控制面 vs 数据面

| 平面 | 手段 | 作用 |
|---|---|---|
| **控制面（相位切换）** | 仅文本标记 `PIGEND` / `PIGFAILED` + **默认边** | 跳转，不注入编排用 tool/MCP |
| **数据面（干活）** | 相位内 **tool loop**（如 search / 读内部源） | 信息搜集等；**不**用于 handoff |

表述约定：

> 编排不注入 tool/MCP；信息类 tool 仍可在相位内使用，且由平台配置。

### 3.3 两标记 + 默认

| 标记 | 含义 | 典型下一跳 |
|---|---|---|
| **`PIGEND`** | 本轮可对外结束 | → **结束 turn** |
| **`PIGFAILED`** | 本轮路径失败，需重规划 | → **清本轮上下文** → **执行前**（带失败路径） |
| **未匹配到合法标记** | **不是错误**，视为默认 | → **loop**（当前默认：回 **执行者**，附带本段输出作反馈） |

解析策略：

- **只正向匹配**约定标记；其余文本一律当默认边。  
- **不存在**「解析失败要报错」的控制路径。  
- 双标记同时出现等冲突：实现时写死优先级（建议 **`PIGFAILED` > `PIGEND`**）。

### 3.4 三相位职责

#### 执行前（Plan / 分流）

- **输入**：`history + 本轮问题 + 额外提示词`（额外提示拼在本轮问题侧）。  
- **额外提示逼出的信息（大意）**：  
  1. 是否简单可直接答？是则最终文本带 **`PIGEND`**  
  2. 是否需要 **外部信息**（项目/环境无关，如网上）  
  3. 应找哪些 **内部信息**（描述）  
  4. 本次回答 **规划**  
  5. 本次回答 **目标**（给执行后当尺子）  
- **相位内**：可 tool loop until idle（预算宜紧、工具宜窄）。  
- **输出**：规划结论（给执行者）；**不含**把执行前完整输入再原样塞给执行者。  
- **路由**：`PIGEND` → 整轮结束（短路）；否则 → 执行者。

#### 执行者

- **输入**：`history + 本轮问题 + 执行前输出`（或执行后无标记时的反馈）。  
- **相位内**：同样 **tool loop until idle**（主信息搜集面）。  
- **输出**：作答 / 中间结果 → **必定**进入执行后。  
- **禁止**输出 /；即使误输出也会被剥离并强制进 POST。结束与返工只由执行后路由。

#### 执行后（一次调用，三逻辑角色同提示词）

同一提示词内兼任：

1. **结果检验（reviewer）**  
2. **目标检验（goal）**  
3. **失败 / 重开发动**  

- **相位内**：可 tool loop until idle（宜只读核对，避免大改目标）。  
- **路由**：  
  - 审查 + 目标 **都过** → 输出 **`PIGEND`** → 结束  
  - 可返工（未过但不致推翻规划）→ **无合法标记** → 默认把本段输出喂回 **执行者**  
  - 路径失败 → **`PIGFAILED`** → 总结经验、**清本轮**、回 **执行前**（history + 问题 + 额外提示 + 失败路径）

### 3.5 状态机（简图）

```text
                    ┌──────────────────┐
                    │     执行前        │  tool loop → idle
                    └────────┬─────────┘
                             │
              最终文本 PIGEND ┼──────────→ 【结束 turn】
                             │ 否则
                             ▼
                    ┌──────────────────┐
              ┌────►│     执行者        │  tool loop → idle
              │     └────────┬─────────┘
              │              │
              │              ▼
              │     ┌──────────────────┐
              │     │     执行后        │  tool loop → idle
              │     │ Review+Goal 一次  │
              │     └────────┬─────────┘
              │              │
              │    ┌─────────┼────────────┐
              │    ▼         ▼            ▼
              │ PIGEND   无合法标记    PIGFAILED
              │    │         │            │
              │ 【结束】  输出作反馈    清本轮上下文
              │              │       失败路径 → 执行前
              └──────────────┘
```

### 3.6 流式与实时

- 每一相位、每一次模型调用、每一次 tool 起止，都可推成 **SSE/事件流**。  
- 对外可伪装成单次 completion/responses 流；对内相位名是否暴露 **随意**（实现自由）。  
- **同模型**多相位：利于缓存、实现简单、延迟可控（相对多模型编排）。

### 3.7 初期能力边界

- **做**：信息搜集、规划、目标检、标记路由、流式包装。  
- **弱化 / 暂缓**：重沙箱、全能 bash、宿主 skill 全家桶默认注入、多模型专家池。  
- **Skill**：机制可保留；API 侧宜 **显式供应**，CLI 可扫本地（与 runtime 默认策略可不同）。

### 3.8 建议钉死的工程护栏（非语义错误，是防死循环）

| 护栏 | 作用 |
|---|---|
| `max_executor_loops` | 执行后默认回执行者的次数上限；超限可强制 `PIGFAILED` 或硬结束 |
| `max_pre_replans` | `PIGFAILED` 回执行前的次数上限 |
| 每相位 `max_tool_rounds` | 防单相位 tool 刷屏 |
| 标记匹配规则 | 如仅匹配独立行/fence；只认合法串 |

这些是 **预算**，不是「解析失败异常」。

### Phase A — 文档与骨架（当前）

- [x] `pigs` 定位为 **native bin**（非 wasm）
- [x] 唯一用户入口：默认本地 API，`pigs --cli -- …` 进 REPL
- [x] 唯一用户入口：默认本地 API，`--cli` 进 REPL  
- [x] 与 `pigs-cli` 二进制名分离（`pigs` / `pigs-cli`）  
- [x] 本文：理解与规划  
- [x] `pigs --describe` / `--once` / 本地 HTTP API
- [x] 相位内核 + 标记路由 + 信息类 tool loop
- [x] 使用 config.local.toml（MaaS）实网验证 `--once` 与 `/v1/chat/completions`

### Phase B — 相位内核（无 HTTP）

- [x] Phase 枚举 Pre/Executor/Post
- [x] PhasedRuntime tool loop until idle
- [x] Marker 解析 PIGEND/PIGFAILED（失败优先；未匹配=默认边）
- [x] 三套相位提示词 + 实网 turn

### Phase C — 与共享 crate 接通

- [x] pigs-llm + pigs-config resolve_model
- [x] 数据面 web_search / web_fetch / internal_notes
- [ ] 服务端多轮会话持久化
- [ ] 相位预算 TOML 化

### Phase D — API 包装

- [x] /health /v1/models /v1/chat/completions（非流式）
- [x] 默认 loopback
- [x] SSE streaming (`stream=true`)
- [ ] 鉴权/配额/超时取消

### Phase E — 可选增强

- [ ] 执行前/后 tool 白名单差异化  
- [ ] 显式 skill 挂载（非扫全机）  
- [ ] 更强执行后端（OS/容器）——与相位机解耦  
- [ ] 轻量多角色 profile（专家团向）——仍同模型相位，不默认上多模型池

## 5. 明确不做什么（近期）

1. 不把 `pigs` 做成 wasm 主交付。  
2. 不用 MCP/tool 名称做相位 handoff。  
3. 不把「标记解析失败」设计成错误通道（只走默认边）。  
4. 不与河豚比模型池；不与 Codex 比完整本机沙箱作为第一优先级。  
5. 不让 API 默认吞入开发者机器上全部 `~/.agents/skills`。

---

## 6. 一句话备忘

**阶段结束看 tool 是否 idle；阶段去向看标记；没匹配到合法标记就默认 loop；同模型三相（后相一次调用兼 review+goal）；控制面两标记，数据面才 tool；pigs 是 bin 运行时，pigs-cli 是本地 REPL，共享库 crate。**
