03 关键设计决策

为什么用非流式？

流式 SSE 里 tool_calls 参数是碎片化的，要按 index 累积拼接。CoreCoder 用了 30+ 行，pigs-llm 更多。教学版用非流式——一次返回完整 tool_calls。

为什么有重试？

429 / 5xx / 网络错误：指数退避重试（1→2→4 秒，最多 3 次）。4xx（非 429）：不重试，请求本身有问题。

为什么在 LLM 层解析 arguments？

API 返回的是 JSON 字符串。在 parse_llm_response 里 from_str，失败降级为 {}，下游工具直接拿 Value——CoreCoder 的设计。

CODE 源码拆解（src/llm.rs）

1. 结构体字段

pub struct LlmClient {
    http: reqwest::Client,  // 120s 超时
    api_key: String,
    base_url: String,
    model: String,
}

字段私有；只能通过 new / from_env 构造。

2. new

reqwest::Client::builder()
    .timeout(Duration::from_secs(120))
    .build()
    .unwrap_or_else(|_| reqwest::Client::new())

LLM 可能很慢 → 120 秒超时。参数 impl Into<String> 可接 &str / String。

3. from_env

变量 必需 默认
OPENAI_API_KEY 是 无 → ConfigError
OPENAI_BASE_URL 否 https://api.openai.com/v1
OPENAI_MODEL 否 gpt-4o

4. chat 主路径

// 1) Message → OpenAI JSON
// 2) body = { model, messages }
// 3) 有 tools → tools + tool_choice: "auto"
// 4) call_with_retry
// 5) parse_llm_response

5. call_with_retry

max_retries = 3; wait_secs = 1;
// POST {base}/chat/completions
// Authorization: Bearer {key}
情况 行为
2xx 解析 JSON 返回
429 或 5xx 且可重试 sleep 后 wait_secs *= 2
其它 4xx 立即 LlmError
网络错误可重试 同上退避
网络错误用尽 NetworkError

退避：1s → 2s → 4s。

6. LlmResponse

pub struct LlmResponse {
    pub content: String,
    pub tool_calls: Vec<ToolCall>,
}

Agent 只看 tool_calls.is_empty()。

7. parse_llm_response

路径：choices[0].message → content + tool_calls。

let arguments: Value = serde_json::from_str(arguments_str)
    .unwrap_or_else(|_| json!({}));
  • 缺字段用 "unknown" / "" / {} 兜底
  • 非法 JSON 参数 → 空对象，不整请求失败
