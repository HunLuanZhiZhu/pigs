Chapter 06 · 核心

Agent 循环

src/agent.rs
这是整个 crate 的心脏。调 LLM → 有工具？执行再循环 : 返回文本。
CORE

所有「智能」来自 LLM。Agent 代码只是忠实执行循环——历史进 LLM，看要干什么，帮它执行工具，把结果再告诉它。

01 12 行骨架

借鉴 CoreCoder agent.py：

# CoreCoder Python
for _ in range(self.max_rounds):
    reply = self.llm.chat(self.messages, self.tools)
    if not reply.tool_calls:
        return reply.text
    results = run_parallel(reply.tool_calls)
    self.messages += results
return "(hit the round limit)"
// 本 crate Rust
for round in 0..self.max_rounds {
    let response = self.llm.chat(&messages, &schemas).await?;
    if response.tool_calls.is_empty() {
        self.messages.push(Message::assistant_text(&response.content));
        return Ok(response.content);
    }
    self.messages.push(Message::assistant_with_tools(...));
    for tc in &response.tool_calls {
        let result = self.tools.execute(&tc.name, tc.arguments.clone()).await;
        self.messages.push(Message::tool_result(...));
    }
}
return Err(MiniAgentError::MaxRoundsReached(self.max_rounds));

03 关键设计决策

为什么 for 而非 loop？

有界循环。可能陷入「工具 → LLM 再要工具」死循环。max_rounds 默认 50（CoreCoder），是保护阀。

为什么先 push assistant 再 push tool_result？

OpenAI 要求每个 tool_call 都有对应 tool 回复。顺序搞反会被 API 拒绝。CoreCoder 的 _answer_pending_tool_calls 就是为中断时补全回复。

工具失败为什么不 panic？

把错误字符串当 tool 结果返回（is_error=true）。LLM 能看到错误并决定下一步——与 CoreCoder _exec_tool 一致。

为什么每次发完整历史？

LLM 无记忆。后果：历史越长越慢越贵。生产级有 auto-compact；教学版不做。

05 用法

let mut agent = Agent::new(llm, tools)
    .with_max_rounds(30)
    .with_system_prompt("你是一个测试 Agent");

let response = agent.chat("帮我创建一个 hello.txt").await?;
agent.reset();           // 清空历史
agent.history_len();     // 不含 system 的消息数

CODE 源码拆解（src/agent.rs）

1. 结构体

pub struct Agent {
    pub llm: LlmClient,
    pub tools: ToolRegistry,
    pub messages: Vec<Message>,
    pub system_prompt: String,
    pub max_rounds: u32,
}

五个字段全 pub，方便教学调试。

2. new

let system_prompt = build_system_prompt(&tools);
Agent { llm, tools, messages: Vec::new(), system_prompt, max_rounds: 50 }

建 Agent 时按当前工具表生成 prompt。之后改 tools 默认不重算（可用 with_system_prompt）。

3. Builder

with_max_rounds / with_system_prompt：消费 self 返回 Self，支持链式调用。

4. 私有辅助

full_messages：system + messages.clone()。system 不存在 self.messages 里。

tool_schemas：self.tools.schemas() 转发。

5. chat 逐步

self.messages.push(Message::user(user_input));

#### 步骤 2：有界 for

for round in 0..self.max_rounds {

#### 步骤 2.1：调模型

let response = self.llm.chat(&self.full_messages(), &self.tool_schemas()).await?;

#### 步骤 2.2：结束分支

if response.tool_calls.is_empty() {

        Ok(o) => (o, false),
        Err(e) => (e.to_string(), true), // 不中断 chat
    };
    self.messages.push(Message::tool_result(&tc.id, &output, is_error));
}

6. reset / history_len

  • reset：清空 messages，保留 system_prompt
  • history_len：只计 messages，不含 system

7. 测试范围

不测真实网络 chat；测创建、builder、reset、full_messages。

8. 心智模型

user push
  → [for]
      full_messages + schemas → LLM
      empty tools? → assistant_text + return
      else assistant_with_tools
           for each tool → execute → tool_result
  → MaxRoundsReached

读完应能默写 chat 分支，并说明为何 assistant 必须先于 tool 入历史。
